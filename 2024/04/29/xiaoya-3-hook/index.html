小雅音箱中间人攻击（三）

X1ngg3
iot iot

发布于：Apr 29, 2024

主动控制与隐私窃取

0x00 前言

在上一篇博客中，通过DNS劫持+流量篡改，我们在攻击服务器上最终得到了设备的osAccessToken，这也是我们唯一得到的有价值的东西，本篇博客将围绕对osAccessToken的利用展开。

在wireshark抓取到的音箱通信流量中搜索osAccessToken，只在最初音箱建立通信的几条http流量中出现相关字段，但都没利用空间。既然音箱自身的流量没有机会，那就对音箱app进行分析看是否有利用空间。音箱作为智能家居的一部分，其手机端app有着自定义对话、查看历史记录、主动播放等功能，如果可以对这些操作进行模拟或者重放的话，那么对用户的隐私数据与音箱的可控性都会造成不小的影响（比如对话记录被窃听和半夜音箱突然自己说话）。

0x01 工具准备

为了能抓取到音箱app的流量，我们需要一台安卓模拟器，最早我是用的mumu模拟器和夜神模拟器这种为游戏专门研发的模拟器，勉强也能用，但是小雅app不知道什么毛病，在模拟器上运行会出现按钮消失的问题，没法进入下一步，然后我转投了android studio（后面省略为as）自带的avd，非常非常的好用，可以按照需求选择安卓版本和API，神众神，秒杀各种安卓模拟器。如果说有缺点的话就是mumu模拟器把证书拖进去直接安装完就可以抓包了，as需要通过adb root来安装，不然默认是安装到用户证书下（如何在as上安装系统证书放在最后了）。

0x02 osAccessToken分析

首先对在攻击服务器上抓取到的osAccessToken进行分析，可以很明显看出来这是base64编码过的，解码后发现这是jwt 格式的，在https://jwt.io/上进行解构分析，头部指明了jwt加密算法采用的是hash 256，jwt类型就是jwt了，载荷部分记录了音箱信息，最后是签名部分的组成。这部分有个了解就行，毕竟我们不打算对jwt本身下手。

JWT的工作流程如下，在本例中，认证服务器为xvstc.xiaoyastar.com，资源服务器为api.xiaoyastar.com。

  • 用户在客户端登录并将登录信息发送给服务器
  • 服务器使用私钥对用户信息进行加密生成JWT并将其发送给客户端
  • 客户端将JWT存储在本地，每次向服务器发送请求时携带JWT进行认证
  • 服务器使用公钥对JWT进行解密和验证，根据JWT中的信息进行身份验证和授权
  • 服务器处理请求并返回响应，客户端根据响应进行相应的操作

下面是对各个字段的分析，在进行字段分析的时候，我们主要关注设备标识和用户校验相关的字段，对于其他固定信息的字段，比如"osClientId":"os.client.000001","productId":"N_PROD1_NANO_TD"，或者app版本和固件版本，比如"appVersion":"3.1.4"、"romVersion":"1.6.96"，我们就不做过多关注了。

session

session部分其实就是一个uuid生成的会话session id，并且通过抓包信息来看，这个session id是由客户端直接生成的，因为服务器返回的session id与客户端请求发起时用的是同一个，并且每个对话的uuid都不同，这样看来，uuid在这里的作用仅仅只有标识唯一对话这一个作用，并不涉及校验。

在jadx中搜索"sid"，找到对应代码hashMap.put("sid", UUID.randomUUID().toString());，这样基本可以确定session就是随机生成的uuid了。

params

params部分大部分数据都是app和音箱的配置信息，这部分信息基本固定，用来标识请求的app版本和音箱型号等。需要注意的是deviceId、osAccessToken、sn三部分，其中osAccessToken、sn都可以在服务器中找到，通过解析osAccessToken可以获取到deviceId，这里需要指出的是，音箱app和音箱自身的deviceID并不相同，中间人攻击抓到的是音箱自身的deviceID。

intent

intent部分主要是当前请求内容，记录了所要请求的服务类型等信息，其中speaker_sn可以通过中间人攻击获得。

sig

签名字段，用来验证请求的有效性和完整性。需要反编译apk去获取签名的生成方法，这个后面会说。

0x03 sig签名算法

这里我使用jadx来进行apk的反编译，这是一款很优雅的软件，直接拖进去就完事了，有作者的博客提到说jadx的反编译结果有时会不够准确或者源代码会对抗jadx，但目前我还没遇到这种情况。

定位关键代码

回到正题来，在https请求的quert string中出现了sig字段，那么我们合理推测，在java应用构建网络请求时直接使用了sig这个字符串，所以我们在通过字符串去定位关键代码的过程中，可以直接搜索"sig"而不是sig，这样出现的结果更加精准。此外，在Android或Java应用中，网络请求通常通过某种形式的HTTP客户端库来处理，如 HttpClient, OkHttp, Retrofit 等，在搜索"sig"字符串出现的结果中，我们可以优先关注包含client、http、request这类字段的包名和类名。

事实上，搜出来的结果第一条就很符合我们的期待，这段代码是为了在执行HTTP请求之前对请求的参数进行排序和签名处理。以下是详细的步骤说明，展示了如何在StringBuffer中构建这个值：

  1. 创建哈希映射：在onPreExecute()方法开始时，首先创建一个HashMap对象来存储请求参数。
  2. 复制请求参数：通过getParamKeyValues()方法获取请求的参数键值对，然后将这些键值对复制到新创建的HashMap中。
  3. 键排序：接下来，获取HashMap的所有键（即参数名），并将它们存储到一个字符串数组strArr中。然后使用Arrays.sort(strArr)对这些键进行字典序排序。
  4. 构建字符串：创建一个StringBuffer对象，然后遍历排序后的键数组strArr。对于每一个键，将键和其对应的值按照”键=值&“的格式追加到StringBuffer中。
  5. 添加固定参数和签名：在所有请求参数处理完后，StringBuffer末尾追加了一个固定的参数"osClientSecret=680d73c11cca46aeb4fd9d28f961dc4a"。然后使用MD5Utils.md5Str32(stringBuffer.toString())对整个字符串进行MD5哈希计算，得到的结果作为签名，添加到请求参数中。
  6. 结束方法：最后，方法通过调用AppMethodBeat.o(107974);进行性能监控，然后方法结束。

签名算法分析

在上述的请求构造过程中，签名是通过MD5Utils.md5Str32(stringBuffer.toString())来实现的，按x进行交叉引用（下意识的按了x，发现jadx中也有这个快捷键，叫做查找用例），点进签名函数进行分析。不进行交叉引用的话，直接搜索类名也可以，鼠标悬停在函数上会显示其所在的类。对于有些函数，直接双击点进去也是可以的。

Java提供了一个名为MessageDigest的类，它属于java.security包。 此类支持诸如SHA-1，SHA 256，MD5算法之类的算法，以将任意长度的消息转换为信息摘要。阅读代码可以发现这里使用的其实就是一个java库中的标准md5，并没有进行魔改，对代码的解释已经标记在注释中了。

0x05 主动控制功能探索

通过对fiddler中抓取到的https请求各字段进行分析，我们发现各字段的值都是可知的，既然请求中的值都是已知的，那是否意味着我们可以通过模拟请求的方式来进行终端伪造，通过模拟音箱app和云端的正常通信过程，控制音箱主动播放特定内容，窃取用户的对话记录等等。

想要实现上述的功能，需要云服务器存在对应的api接口，反映在app上，就应该有对应的功能按键。之前做小爱音箱的时候，小爱音箱是可以直接播放指定歌曲或者播放指定文本的，所以我下意识的就认为这功能大家都有，但是小雅app只能主动控制播放喜马拉雅平台上的各种有声读物，并不能直接控制播放指定歌曲或者特定文字。

虽然不能直接控制，但是在小雅app中，有个自定义AI口令的功能，设置自己的AI口令后，通过指令词可以执行特定的任务，比如让音箱说出指定的文字，播放特定音乐等。

此外，可以在app上可直接输入文本与小雅对话，小雅的回答会通过音箱播放，结合我们的自定义AI口令功能，可以做到控制音箱任意播放的效果。例如输入11后，小雅回答了我们指定的回答文本。

但这样操作难免会留下痕迹，我们希望我们的行为是无法溯源的，至少不能留下这么明显的痕迹，所以在创建+唤醒后，我们还应该删除这条指令。额外的，在和音箱对话的界面，还隐含的查询历史记录的功能，当我们打开对话界面的时候，首先进行的是加载历史对话记录。

0x06 请求伪造原理

到目前为止，我们似乎已经拥有了进行请求模拟所需要的全部内容，我们已经分析完了一次https请求所需要的所有参数，并且构造出了音箱主动播放的控制链，下面要做的就是构建请求了。但在这之前，我对于请求构建仍有着一些疑惑（这主要受限于我个人水平），比如服务器是如何验证发起请求的用户身份的，是否存在更深层次的校验，为什么一行简单的代码即可完成https请求过程。

https请求是如何构建的

首先是为什么在python的requests库，通过https来请求服务器资源看起来如何简单，一行代码即可轻松搞定，毕竟这是https，通信前是需要进行握手的。

但实际上，为了完成这条请求，requests库做了：

  1. 建立连接：requests.get()函数将尝试与api.ximalaya.com上的HTTPS端点建立连接。这个过程包括DNS解析，找到api.ximalaya.com对应的IP地址，并与之建立TCP连接。
  2. SSL握手：一旦TCP连接建立，requests库将执行SSL握手过程，与服务器端建立安全连接。这个过程包括协商加密算法、验证服务器证书等。
  3. 发送请求：一旦SSL握手成功，requests库将发送HTTP GET请求到https://api.ximalaya.com/smart-os-gateway/app/invoke这个URL，同时将参数param作为查询参数传递给服务器。
  4. 接收响应：服务器收到请求并进行处理后，会返回HTTP响应。requests库将接收到的响应解析为一个Response对象，其中包括状态码、响应头和响应体等信息。
  5. 断开连接：通信完成后，客户端和服务器会断开连接，释放资源。

这个过程中，SSL握手确保了通信的安全性，而requests库封装了这些细节，这让我们可以通过简单的API发送HTTPS请求，而无需手动处理底层的连接和加密操作。

如何验证请求者

其次是服务器是如何验证发起请求的用户身份的，这个问题可以换一个说法，即服务器是如何识别非自己的客户端调用API的。回答是很难，在本例中，通信架构为C/S架构，服务器的代码我们无法得知，但是客户端一旦发出去就不可控了，我们可以随意对这个客户端（在这里是小雅APP）进行逆向分析，事实上，这也正是我们做的工作。

下面是最基础的app与服务器的加密与认证逻辑：

  • 每个app都有一个app_id和app_secret，app_id可以公开，app_secret必须保密，但是也必须放在app的代码里面。app_secret的保密性就是最关键的环节，在本例中，app_id可理解为deviceId，app_secret可理解为osClientSecret。
  • 每次app发出一个请求的时候，都需要对数据进行签名。大致就是综合数据、app_id、app_secret和时间戳进行hash运算，这个环节可以参考上述的小雅app的sig生成方式。然后将这个值、app_id、时间戳和数据一起发送，放在url参数里面或者http头里面都可以的。
  • 服务器收到请求之后，首先看对比时间戳和当前时间，如果超过某个允许的延时时间间隔，比如说20秒钟，认为这是一个”旧的”请求，就拒绝接受。可能是请求被黑客截获了，过了一段时间又重放出去的。如果时间戳校验没问题，接下来就是按照相同的规则进行签名，看两个签名是否的是一致的。如果是一致的就说明数据没有被篡改或者数据不是没伪造的。因为黑客不掌握app_secret的情况下，是不可能伪造签名的。
  • 有时候为了更加的安全，会将url也加入hash中，这样可以防止数据的目的地被篡改。还有一个是在数据中再加入一个nonce字段，每次都是随机生成，认为不会重复，这样在服务器允许的延时时间间隔中也只能使用一次了，没有nonce的时候，服务器允许时间间隔内数据也是可以使用多次的。

上面的防范是在app_secret保密的情况下，如果app被反汇编获取到了app_secret，那就无能为力了。当然这个一个对抗的过程，app厂商是可以无限的增大破解者的难度的，比如：

  • 限制每个app_id的api调用频率，超过频率禁用掉或者需要验证码
  • 使用私有通信协议
  • 对app代码进行混淆，封装成.so机器码等
  • 使用ssl以及对数据进行非对称加密
  • 对识别出来的非法调用，延时返回假数据

如何保证请求的安全

在本例中，服务器使用使用token+签名认证来保证请求安全性。

token+签名认证的主要原理是：

  • 做一个认证服务，提供一个认证的webapi，用户先访问它获取对应的token
  • 用户拿着相应的token以及请求的参数和服务器端提供的签名算法计算出签名后再去访问指定的api
  • 服务器端每次接收到请求就获取对应用户的token和请求参数，服务器端再次计算签名和客户端签名做对比，如果验证通过则正常访问相应的api，验证失败则返回具体的失败信息

安全的关键在于参与签名的TOKEN，整个过程中TOKEN是不参与通信的，所以只要保证TOKEN不泄露，请求就不会被伪造。然后通过timestamp时间戳用来验证请求是否过期，这样就算被人拿走完整的请求链接也是无效的。Sign签名的方式能够在一定程度上防止信息被篡改和伪造，保障通信的安全。但我们一是通过中间人攻击获取了音箱osAccessToken，二是通过反汇编获取了app的osClientSecret，这两个要素使得我们伪造请求变得可行。

构建请求

在知道了原理以后，这部分内容就很简单了，根据fiddler中抓取到的数据，在python中构造出对应的结构体，然后通过requests库请求就完事了。

0x07 android emulator

step1 选择安卓模拟器

这里我使用的安卓模拟器为android studio，版本为Android Studio Iguana | 2023.2.1 Patch 1。作为一款转为Android应用开发而设计的IDE，它自带的模拟器对开发和测试非常友好，不会像网上那些为游戏而设计的安卓模拟器一样总有各种奇怪的bug（比如吞按键。）

在android studio创建镜像时，会有选择设备型号和安卓版本/API的流程，比如我的需求是模拟一台手机，所以我选的是Phone-pixel 3a。

  3. 以root身份连接adb：adb root

  4. 禁用安全启动验证：adb shell avbctl disable-verification

  5. 重启设备：adb reboot

  6. 以 root 身份重新启动 adb：adb root

  7. 以读写方式重新挂载分区：adb remount（如果执行完后系统重新启动，请再次重新启动adb reboot，并再次运行adb remount。）

  8. 推送重命名后的fiddler证书：adb push <path_to_certificate> /system/etc/security/cacerts

  9. 设置证书权限：adb shell chmod 664 /system/etc/security/cacerts/<name_of_pushed_certificate>

  10. 重启设备：adb reboot

此外，在安装结束后，如果你想使用安装的系统证书的话，也必须通过-writable-system选项来启动模拟器！！！
